1 The 2-Entity Consumer Model — Samajhna Zaroori Hai

Yeh do alag database entities hain. Inhe samjhe bina koi bhi screen galat ban sakti hai.

👤 Consumer — Platform Profile

Unique by: phone number
Yeh person hai — naam, phone, email, default address. Platform pe ek baar banta hai. Agar Vendor A ne Raju ko add kiya aur Vendor B ne bhi Raju ka same phone diya, toh same Consumer record use hoga — naya nahi banega.

Status: PENDING (invite sent) → ACTIVE (accepted) | REJECTED (declined)
Origin: VENDOR_CREATED (is guide ka flow) | SELF_REGISTERED (future consumer app)

🔗 ConsumerVendor — Relationship

One record per: Vendor × Consumer × Capability × Address
Yeh connection hai — kaun vendor, kaun consumer ko, kis address pe, kaun si service (MILK_DELIVERY, TIFFIN etc.) de raha hai. Same consumer ke multiple relationships ho sakte hain.

Status: PENDING → ACTIVE → BLOCKED / DISCONNECTED
Holds: capabilityId, addressId, preferredDeliverySlot, deliveryInstructions

💡
Real Example: Raju (9876543210) → 1 Consumer record. Raju milk Sharma Dairy se leta hai, water AquaPure se, tiffin FreshBox se → 3 ConsumerVendor records, teeno ek hi Consumer se link. Yeh intentional hai.

2 S7 — Vendor Adds Consumer: Complete Decision Tree

Ek hi API call hai — POST /vendors/{id}/consumers — lekin backend internally 3 alag scenarios handle karta hai. UX ko yeh teeno dikhana chahiye.

Backend Logic — addConsumerByVendor()
Vendor calls POST /vendors/{id}/consumers
{ phone, name, capabilityId, category, addressId, slot, instructions }
↓
Backend checks: findByPhoneAndActiveTrue(request.getPhone())
📱 Consumer FOUND (existing.isPresent() = true)
Same phone already on platform. isNew = false.
Consumer record is reused.

Then checks duplicate relationship:
existsByConsumerIdAndVendorIdAndCapabilityIdAndAddressId

→ If SAME vendor + SAME capability + SAME address: 409 DUPLICATE error
→ Otherwise: new ConsumerVendor created + WhatsApp sent
|
🆕 Consumer NOT FOUND (existing.isPresent() = false)
Brand new person. isNew = true.
New Consumer record created:
• status = PENDING
• origin = VENDOR_CREATED
• onboardedBy = vendorId
• language = "hi" (default)

Then: new ConsumerVendor created
Then: CONSUMER_CREATED event published
→ WhatsApp invite sent to consumer

🆕 Path A: New Consumer

Phone number pehle kabhi platform pe register nahi hua.

  • Consumer record banega (PENDING)
  • ConsumerVendor record banega (PENDING)
  • CONSUMER_CREATED event fire hoga
  • WhatsApp invite jayega
✅
UX: Show "Consumer added! Invite sent." success state

🔗 Path B: Existing Consumer

Phone already on platform — kisi aur vendor ne add kiya tha ya self-registered hai.

  • Consumer record naya nahi banega
  • Sirf naya ConsumerVendor banega
  • CONSUMER_CREATED event nahi fire hoga
  • WhatsApp invite jayega is vendor ke liye
💡
UX: Search screen pe "Consumer already on platform" card dikhao, phir proceed karwa

🚫 Path C: Duplicate

Same vendor + same consumer + same capability + same address. Already serving!

  • Backend throws DUPLICATE_RESOURCE error
  • HTTP 400 response
  • Kuch naya nahi banega
⛔
UX: "Already linked! Aap pehle se yeh service is address pe de rahe hain."
⚡
Allowed: Same consumer, different scenario. Same consumer + same vendor + different capability (e.g., milk aur water dono) = allowed. Same consumer + same vendor + different address (ghar aur office) = allowed. Backend check sirf exact 4-way match karta hai: vendorId + consumerId + capabilityId + addressId.

4 Planned Consumer Creation Flows (Backend mein abhi nahi hain)

Yeh flows backend mein currently implement nahi hain lekin roadmap mein clearly chahiye. Har flow ke liye backend improvement needed hai.

⚠️
Current State: Abhi sirf ek hi vendor-side creation flow hai — manual phone input (POST /vendors/{id}/consumers). Niche likhe flows planned improvements hain — inhe UI mein "Coming Soon" ya separate development items ke roop mein treat karo.
📋 CSV Bulk Import

Use case: Vendor ke paas 50+ customers hain jo ek register mein hain. Ek CSV upload se sab add ho jayein.

Expected behavior: CSV mein phone, name, address, slot columns → backend loop karke har row ke liye addConsumerByVendor() call kare → existing consumers auto-detect karke sirf relationship banaye → bulk result (success count + error rows) return kare.

API needed: POST /vendors/{id}/consumers/bulk (multipart CSV ya JSON array)
Backend status: Not implemented — needs new endpoint + CSV parser + batch transaction

📦 First-Order Auto-Consumer Link

Use case: DYNAMIC_PRICING vendors ke liye (laundry, cleaning etc.) — vendor directly ek order create karta hai kisi ke liye bina consumer onboarding ke. Order creation pe consumer auto-create hona chahiye.

Expected behavior: Vendor creates order → if no consumerId but phone given → backend consumer lookup → create if new → link to order. Consumer baad mein WhatsApp pe invite dekhta hai.

API change needed: VendorOrderController mein phone-based consumer auto-create logic
Backend status: Not implemented — FIRST_ORDER_FREE enum sirf discount type hai, consumer creation nahi

🎙️ Voice Dictation Add

Use case: Vendor subah 5 baje milk route pe hai, haath geela hai. "Add Raju, 9876543210, Baner, milk, 6 to 7" bol ke consumer add kare.

Expected behavior: Flutter speech-to-text → parse phone/name/service/slot → pre-fill form → vendor confirms → same POST /consumers API call.

Backend change needed: No backend change — sirf Flutter speech-to-text integration
Status: Flutter work only — backend already handles the request

📷 OCR from Written Register

Use case: Vendor ke paas purana haath ka register hai — Raju, 9876543210, Ghar, 2 litre milk, daily — woh photo khichta hai aur sab auto-add ho jata hai.

Expected behavior: Camera → ML Kit / Tesseract OCR → structured data extract → bulk confirmation screen → vendor verify karke sab approve → bulk POST to API.

Backend change needed: Bulk endpoint zaroor chahiye (CSV bulk se same)
Status: Flutter ML Kit + bulk API endpoint — dono needed

🎯
Developer Note: In flows ke liye sabse pehle Bulk Create API banana zaroori hai — CSV, OCR, aur voice teeno ke liye base yahi API hogi. Ek hi endpoint: POST /vendors/{id}/consumers/bulk jo List<VendorAddConsumerRequest> accept kare aur BulkCreateResult (success[], failed[]) return kare.

Validation Rules for CSV Import
  • Phone: exactly 10 digits, starts with 6-9 (existing regex)
  • Duplicate phone in same CSV: warn, skip second occurrence
  • Existing consumer (already on platform): reuse — create only relationship
  • Max 500 rows per import (to prevent abuse)
  • Address: if address_text given, geocode attempt + create Address entity via address module
  • Product name: fuzzy match against vendor's active products catalog
